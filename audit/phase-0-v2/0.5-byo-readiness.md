# Phase 0.5 — BYO Mode Readiness

**Date:** 2026-04-25

## Two distinct concepts

The codebase already has "BYO-like" capability in the form of **vendor
report imports**. v2 §9 BYO is a **different** concept: substitute-for-
preflight pre-computed analysis JSON. Both should coexist; both are
toggleable per the §16 cascade. **Don't conflate them.**

| Concept | Purpose | Endpoint | Schema | Status |
|---------|---------|----------|--------|--------|
| Vendor-report import | Bring your existing PitStop/Callas/Acrobat/etc. report into a lintPDF job for unified delivery + viewer + sharing | `POST /api/v1/jobs` with `external_format=...` form field | Each parser at `imports/{vendor}.py` returns native finding rows | **shipped** |
| **v2 §9 BYO** | Skip lintPDF preflight entirely; tenant's RIP / pre-flight has already produced separations + heuristics; lintPDF turns the JSON into a viewer + decisions + report | **does not exist** — proposed `POST /v1/byo/sessions` | §9 JSON Schema (separations, rich_black, content_heuristics, precomputed_deltas, dieline, etc.) | **greenfield** |

## What v2 §9 BYO requires (greenfield)

Per playbook §9 + Phase 2.EPM.5:

### Endpoint
- `POST /v1/byo/sessions` — accept §9-compliant JSON payload; return session ID
- `GET /v1/byo/sessions/{id}` — viewer hydration data + report
- `GET /v1/byo/sessions/{id}/decisions` — operator decisions (parity with native job)

### Schema (§9 JSON Schema)
Required keys:
- `schema_version: "1.0"`
- `document` — filename, sha256, page_count, pdf_version, pdfx4_compliant, output_intent, source_profile_hint, trapped_flag
- `press_target` — model, mode (EPM/EPM+/CMYK/IndiChrome/Custom), substrate_class, alwan_hpm_enabled, device_link_id
- `page_boxes[]` — per-page Media/Crop/Trim/Bleed/Art + rotate
- `fonts[]` — name, subtype, embedded, subset, has_to_unicode
- `separations` — C/M/Y/K {coverage_pct, max_pct, area_gt_zero_pct, distinct_tints, histogram_16}
- `separations.spots[]` — name, alt_space, alt_values, lab, coverage_pct, is_technical, iso_19593_group, iso_19593_type
- `rich_black` — recipe_min, area_pct, pages_affected[]
- `total_ink_coverage` — max_pct, above_260_pct_area, above_320_pct_area, histogram[]
- `content_heuristics` — skin_tone_pixel_pct, deep_shadow_area_pct, neutral_gray_area_pct, small_black_text_count, thin_black_line_count, registration_color_in_artwork
- `precomputed_deltas` (EPM-Advanced) — delta_e_above_threshold_pct, delta_c_above_threshold_pct, heatmap_png_url
- `barcodes[]` — page, bbox, symbology, value, x_dimension_mm, magnification_pct, iso_15416_grade, iso_15415_grade, quiet_zone_*
- `dieline` — detected_via, spot_names[], path_data_present, is_closed, self_intersects, z_order, set_to_overprint, tint_pct
- `thumbnails[]` — page, composite_png_url, separation_pngs{}

### Validation
- JSON Schema validator (Pydantic + jsonschema), versioned at `/byo/schemas/byo-payload.v1.json`
- Reject malformed payloads with field-level errors (per playbook §11 stop condition)

### Per-check BYO mode declaration
Per playbook §2.1 (per-check design note): every user-facing check declares
which §9 keys it consumes and what degraded behavior it has when keys are
missing. Two practical modes:
- **Full mode** — all required keys present; check produces full finding
- **Degraded mode** — partial keys; check produces aggregate page-level
  finding (or skip with reason recorded)

### Viewer hydration parity
Per playbook §13.1 + §13.6: identical overlay UX for BYO sessions.
- If BYO supplies geometry (bbox / polygon), use it
- If BYO supplies only metrics, render aggregate page-level annotation
- Drawer header shows "BYO" badge so operator knows source

| Check family | Required §9 keys | Degraded behavior |
|--------------|------------------|-------------------|

| B-* barcodes | `barcodes[]` | Full BYO support |
| EPM-* | All EPM tier keys (separations, rich_black, content_heuristics, precomputed_deltas) | Score with evidence-weight penalty if partial |

## EPM-specific BYO behavior

Per playbook §2.EPM.5:
- **EPM-Core (rule-based):** requires `separations`, `separations.spots[]`,
  `rich_black`, `total_ink_coverage`, `content_heuristics` (subset),
  `document`, `press_target`. Score using deterministic formula.
- **EPM-Advanced:** add `content_heuristics` (skin_tone, deep_shadow,
  neutral_gray) + `precomputed_deltas`. Without these, treat features as
  null (don't penalize as 0); apply evidence-weight penalty per §2.EPM.4
  ("Qualified on provided data — confirm with full preflight").
- **EPM-AI-Explain:** uses Advanced + LLM emits 1–3 sentence explanation;
  failover to deterministic templated string if LLM unavailable (per cost
  cap or rate limit).

## Dependencies

| BYO §9 deliverable depends on | Notes |

| Tier-0 primitives | If a check operates in degraded BYO mode, the predicate stack should be the same; primitives unify the path |
| Wave V V-09 (BYO viewer hydration) | Viewer needs to render BYO-only sessions |
| Toggle registry (§16) | Per-check BYO mode flag (full / degraded / disabled) |

## Recommendation

**BYO §9 ships with Phase 2.EPM** (per playbook sequencing — §5 / §17.1).
EPM is the canonical first consumer; ship the schema validator, endpoint,
and viewer hydration as part of EPM Core. Other checks declare BYO support
incrementally during Wave D / Wave C. Don't gate BYO behind universal check
support.

Do NOT conflate vendor-report import and BYO §9. Different surfaces:
- `POST /v1/jobs` with `external_format=...` — vendor report (continues to ship)
- `POST /v1/byo/sessions` — pre-computed JSON (new)

Both are toggleable per tenant via §16 cascade.
