# Phase 0.6 — Viewer & Overlay Infrastructure

**Date:** 2026-04-25
**Cross-references:** Phase 0.1 §G; Phase 0.4 Wave V touchpoints.

## 0.6.1 Viewer entry point

**Component:** `packages/app/components/viewer/PdfViewer.tsx` (1,352 LOC).
**Tech: Canvas + pre-rendered PNG tiles** — NOT pdf.js, NOT mupdf-wasm.
**Tile source:** `/api/v1/viewer/tiles/{jobId}/{page}.png` rendered by engine `Worker-Tiles` (Pillow + pdf2image/poppler).
**Coordinate origin:** PDF user space (1/72"), per-page; pixel mapping via `ptsToPixels = 1.0 + (tileDpi - 72) / 72` (PdfViewer.tsx:532). Default DPI 150.
**Page rotation / UserUnit:** handled per-tile by engine renderer; client receives pixel-space tiles.

**Implication for v2:** Wave V cannot rely on client-side PDF parsing. Every overlay must be bbox-driven; geometry is server-supplied via finding records and viewer config.

## 0.6.3 Coordinate system

| Layer | Space |
|-------|-------|
| Engine emit (finding `bbox`) | PDF user space (1/72") |
| API JSON | PDF user space |
| Tile PNG | Pixel space (DPI-dependent) |
| Canvas overlay rendering | Pixel space — converted via PdfViewer's `ptsToPixels` factor |
| Operator click input | Pixel space — converted to user space for storage |

CTM, page rotation, and UserUnit are **server-side** baked into the tile.
The client treats the tile as a flat image with a known points-per-pixel
factor. This is convenient — overlay geometry is bbox-driven and invariant
to PDF transformation matrices — but means **per-finding click location**
needs to round-trip user-space ↔ pixel-space carefully.

## 0.6.4 Click / hover interaction

**Current pattern:**
- `PageCanvas.tsx` registers a `pointerdown` handler on the canvas
- Hit-test via bbox intersection in pixel space
- On hit → `onFindingClick(findingId)` callback → `FindingsPanel` selection
- **Limitation: only one finding per click** — no per-instance vs per-type distinction; no drawer-anchored-to-bbox UX

**v2 §13.4 + §13.5 requirements:**
- Multiple findings stack at the same location → show count badge → drill-down list view
- Click → drawer panel anchored to annotation (not global sidebar)
- Drawer contains: header, evidence, remediation, decision buttons, note field, decision history
- Type-level vs instance-level distinction with drill-down

**Wave V V-03 deliverable scope:**
- Refactor `PageCanvas.tsx` hit-test to return all overlapping findings, not just the topmost
- Build new `FindingDrawer.tsx` component anchored to viewport (positioned relative to clicked annotation)
- Extend `BoxOverlay.tsx` to render count badges when findings stack

## Risk callouts for Wave V planning

1. **Drawer-anchor positioning** is non-trivial in mobile — viewport detection, scroll-locking, etc. Reuse `MobileDrawer` patterns.
2. **Per-finding click hit-test** must perform on stacked findings without lag; `PageCanvas.tsx` rendering loop already does compositing — extend the hit-test, don't re-implement rendering.
3. **Tile re-rendering** on layer toggle currently triggers full server round-trip. Severity sub-toggles (info/warn/error) should be **client-only** to avoid round-trip cost.
4. **Theming migration** must preserve current colors as defaults — tenants who haven't customized see no change.
5. **Decision persistence latency** — tRPC mutation should be optimistic with rollback on failure; otherwise operator click feels laggy.
