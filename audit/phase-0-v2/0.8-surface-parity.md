## Toggle ID stability across surfaces

§16.1 hard requirement: toggle IDs are stable across API / SDK / desktop.
Today:
- **Check IDs (LPDF_*):** ✅ stable; sourced from `check-catalog.json` shared between app + engine
- **Profile IDs:** ✅ stable; engine canonical
- **Layer IDs (viewer):** ✅ stable; defined in viewer types.ts; same in desktop viewer types
- **Webhook event IDs:** ✅ stable; defined in engine + Prisma enum

**v2 additions:**
- Toggle IDs for new categories, tier groups, epm_module, epm_thresholds — all greenfield
- Per-tenant icon set IDs (V-11) — namespaced per tenant
- Decision event IDs (§14) — defined in engine

**Recommendation:** Centralize toggle IDs in a single TypeScript types file
+ a Pydantic enum, both auto-generated from a single source-of-truth JSON
manifest. Prevents drift.

## Hardest cells

The cells that will take the most engineering effort:
1. **Per-check toggle from desktop** (V-07 + V-13). Today, tenant must
   fork a profile to disable one check. v2 wants direct toggle. This means
   surfacing the check catalog in desktop UI, with per-check enable/disable
   that overrides the workflow profile.
2. **EPM thresholds editing on desktop** (EPM + V-13). 14+ thresholds need
   first-class UI in desktop's workflow-edit panel.
3. **Audit log read with surface attribution** across all three. Different
   query semantics on each surface; same ToggleAuditLog table feeds them.
4. **Toggle registry read endpoint** must remain stable as new categories
   land. Versioned endpoint (`/v1/toggles?schema_version=1`) recommended.

## Recommendation

Wave V V-07 (resolver) + V-08 (audit log) + V-12 (migration) **must** ship
before any new check or threshold introduces a configurable knob. Otherwise
each new toggle inherits the legacy inconsistent foundation and the
remediation cost grows.
