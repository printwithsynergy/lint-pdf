# Phase 0.7 — Config Cascade Audit

**Date:** 2026-04-25
**Cross-references:** Phase 0.1 §H; Phase 0.4 Wave V touchpoints.

## 0.7.1 Audit answers (20 questions per playbook)

### Tenant level (Q1–3)

**Q1. Where is tenant-level config stored? Schema? Where read? Where written? Cache?**
- Stored in Prisma `Tenant.settings` (JSONB column, no schema enforcement) +
  engine `Tenant` SQLAlchemy table for entitlements
- Schema: open JSON; convention-based fields; entitlement fields enforced
  via `TenantEntitlements` Pydantic model in `tenants/entitlements.py`
- Read: tRPC `tenant.getConfig()` (app); `auth.dependencies.get_tenant()` (engine)
- Written: tRPC `tenant.updateConfig()` (app); engine has no direct write path
- Cache: client-side React state only; no server-side cache

**Q2. Knobs exposed today.**
Open JSON allows arbitrary fields. Conventions found:
- `branding.{primaryColor, emailButtonColor, loginBgColor, ...}` (Pixie Dust)
- `aiCredits.monthlyLimit` (entitlement merged from plan tier)
- `webhooks[]` (Webhook model — first-class table)
- `customProfiles[]` (CustomProfile model — first-class)
- `endpoints[]` (Endpoint model — first-class)
- `approvalChains[]` (ApprovalChain model — first-class)

There is **no enumerated knob registry**. Anyone can add a key to
`Tenant.settings`. v2 §16 registry is greenfield.

**Q3. Tenant-level audit log of config changes?**
**Absent.** No `ConfigAudit` table; no middleware on tRPC update mutations.

### Workflow level (Q4–7)

**Q4. Workflow as first-class entity?**
**No.** Codebase has these intermediate scopes:
- `Profile` (CustomProfile / SystemProfile) — preflight rule bundle
- `BrandSpec` — branding override
- `ApprovalChain` — accept/reject workflow
- `Endpoint` — webhook destination
- `ImportMapping` — vendor-format mapping

None of these is a *composite* "workflow" entity that bundles profile + branding + endpoint + approval as one inheritable layer between tenant and per-call.

**Q5. What does workflow inherit/override (if it existed)?**
N/A — does not exist.

**Q6. If it doesn't exist, what intermediate scoping is there?**
- `Profile` overrides preflight rules per-call
- `BrandSpec` overrides branding per-call
- `Endpoint` is referenced per-call but stored at tenant level
- `ApprovalChain` is selected per-call

These are **independent** scopes — no single composite layer bundles them.

**Q7. Multiple workflows per tenant? Shareable?**
N/A. Profiles are user-creatable and not shareable across tenants today.

### Per-call level (Q8–10)

**Q8. Request-time overrides accepted by HTTP API.**
`POST /api/v1/jobs` form fields:
- `profile_id` (overrides tenant default profile)
- `external_format` (vendor report import)
- `brand` (default / anonymous / lintpdf / profile)
- `brand_profile_id` (when brand=profile)
- `jdf_companion` (base64 JDF sidecar)
- `approval_template_id`
- `file`, `filename`, `metadata` (form data)

Headers: `x-api-key` (tenant auth), `content-type` (form-data).

Other endpoints (batch, brand_specs, profiles, endpoints) accept similar but distinct override surfaces.

**Q9. SDK exposes same overrides as HTTP API?**
SDK at `packages/sdk-python/`. Confirmed parity for `POST /v1/jobs`. Other endpoints not all surfaced through SDK; gap to be enumerated by Phase 1.

**Q10. Desktop exposes same overrides as API/SDK?**
Yes for job submission via FolderConfig (mirrors Job request fields). Other endpoints (brand specs, approval chains) editable via the dashboard, not the desktop app. Desktop is read-mostly for non-job config.

### Cascade resolution (Q11–13)

**Q11. Tenant says X, per-call says Y — who wins?**
Per-call wins for `profile_id`, `brand`, `external_format`, etc. Documented
in CLAUDE.md §"Per-call overrides"; no in-code documentation. **Not
consistently enforced** — some endpoints silently merge tenant defaults;
others fully replace.

**Q12. Lockable / enforced toggles?**
**Absent.** No lock semantics; any per-call request can override any tenant
default.

**Q13. Partial override resolution?**
- Profile: override fully replaces tenant default profile
- BrandSpec: override fully replaces
- ApprovalChain: override fully replaces
- AI features: set-union merge in `TenantEntitlements`

**Inconsistency:** Different fields use different merge strategies. v2
§16's resolver needs explicit per-toggle declaration.

### Surface parity (Q14–16)

**Q14. Desktop shares same canonical store as API?**
**Yes** — desktop is HTTP client; FolderConfig is local mirror; server is canonical.

**Q15. Desktop sync mechanism?**
On startup: API call to fetch tenant config + folders. No reconcile-on-reconnect protocol; offline mode unsupported. **Wave V V-13 greenfield.**

**Q16. Toggle ID consistency across API + desktop?**
- Check IDs (LPDF_*): consistent (catalog.json shared)
- Profile IDs: consistent (engine canonical)
- No drift observed at the ID layer

### EPM specifically (Q17–19)

**Q17. EPM settings configurable at any of three levels today?**
- Rich-black recipe: hardcoded in `advanced_color_analyzer.py` (target C 60% / M 40% / Y 40% / K 100%)
- TAC limits: hardcoded per-substrate constants
- ΔE thresholds: not implemented (no device-link)
- LLM provider/model: hardcoded per analyzer (Haiku 4.5 for audit, Sonnet for vision)
- LLM cost cap: per-tenant via `monthly_ai_credits` entitlement; no EPM-specific channel
- EPM module tier: not toggleable today (one analyzer)

**All EPM thresholds are hardcoded.** v2 §2.EPM.8 requires per-tenant
override + per-workflow override + per-call override.

**Q18. EPM in same config substrate?**
EPM analyzer is in the same code substrate as other analyzers. No
EPM-specific config table; settings are constants in code.

**Q19. EPM AI-Explain LLM cost cap per-tenant or global?**
LLM cost cap is **per-tenant** via `TenantEntitlements.monthly_ai_credits`,
but **not EPM-specific**. v2 §15 / §17 require EPM-AI-Explain to be off by
default until tenant sets a cost cap; today, AI calls deduct from the
shared monthly_ai_credits pool with no separate EPM channel.

### Audit & compliance (Q20)

**Q20. Every config override audit-logged with `{tenant, workflow, call_id, override_path, value, set_by, set_at}`?**
**No.** No audit table; no audit middleware. Per-call overrides are
captured in the Job record (which references profile_id, brand, etc.) but
this is data, not an audit log; mutations to tenant config are
unobserved.

## 0.7.3 Desktop app surface audit

Per playbook §0.7.3 questions:

**Tech stack:** Tauri 2.2 + Vite 6 + React 19. Native binary on macOS (universal), Windows (.exe with .ico), Linux (AppImage).

**Authentication:** API key (`x-api-key` header). User pastes their tenant API key into Settings; key persists in Tauri's `@tauri-apps/plugin-store`. No OAuth, no SSO.

**Config sync (server → desktop):** On app launch, desktop fetches tenant config (folders, profiles, endpoints) from server and stores in local `AppConfig`. **No periodic resync; manual reload required for new folders / profiles.**

**Config sync (desktop → server):** When user creates / edits a folder in desktop, it POSTs to server. Server is canonical.

**Offline mode:** **Unsupported.** Every preflight requires server round-trip. If server is unreachable, desktop displays error; queued submissions are not retried.

**Reconcile-on-reconnect:** Not implemented. If desktop edits a folder offline (impossible today since config UI requires server), there's no merge protocol.

**Embedded engine:** **No.** Desktop is HTTP client only. All preflight runs server-side. No local pikepdf, no local rasterization.

**Wave V V-13 deliverable scope:**
- Add periodic resync (e.g., every 30s while app focused)
- Add **read mirror** for offline use: if config fetch fails, fall back to last-known-good cached config; gray out edit UI; show "offline mode" banner
- Add reconcile protocol for the (future) case where offline edits are allowed
- Audit-log surface attribution: any config write from desktop tagged with `surface: "desktop"` so the audit log distinguishes from API / SDK
- Toggle ID consistency: confirmed today; ensure the new ToggleRegistry surfaces the same IDs to desktop (preferably via `/v1/toggles` endpoint)
